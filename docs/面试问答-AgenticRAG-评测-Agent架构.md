# 面试问答：Agentic RAG、评测方法、Agent 架构

> 基于 gakki-ai-fitness 项目实践的回答，2026-07-07

---

## 一、Agentic RAG vs 普通 RAG

### Q3: Agentic RAG 和普通 RAG 对文档处理有什么区别？

本项目里两者对文档的处理方式不同：

**普通 RAG**：文档一次性入库。切 chunk → embed → 存向量库。查询时直接用 query 向量做相似度检索，返回 top-K chunks。文档是"死"的，不会根据查询动态调整。

**Agentic RAG**：文档处理是迭代的。本项目的 AgenticRAG 在检索后多了一步 LLM 自评——"检索结果够不够回答这个问题？"。不够的话，LLM 改写 query 重新检索，最多 3 轮。每一步 LLM 都会判断"还需要文档的哪个部分"，相当于动态决定文档的哪些片段值得看。

核心区别：普通 RAG 文档是提前切好的固定窗口，Agentic RAG 文档是 LLM 按需"导航"到的。

### Q4: Agentic RAG 的终局是什么？

从大视角看，Agentic RAG 的终局不是"更好的检索"，而是**检索和推理的边界消失**。

当前架构：检索器 → 返回文档 → LLM 读文档 → 生成回答。检索和推理是分离的两个步骤。

终局形态：LLM 在生成过程中自主决定"我现在需要查什么"，像人在想问题时翻书一样。不是先搜再读，而是边想边搜、搜完再想、想了再搜——**工具调用成为推理过程的一部分**，不再是一个独立的 pipeline 阶段。

本项目已朝这个方向走了一步（LLM 自评 + 改写查询），但本质上还是单轮检索-生成循环，离真正的推理-检索交织还有距离。

### Q5: 数据的终局是什么？

健身领域数据的终局是**结构化知识图谱 + 非结构化文档的融合**。

本项目已做雏形：动作库（结构化，39 个动作的属性/肌肉/器械） + 知识库（非结构化，18 篇文档，46 chunks） + Neo4j 知识图谱（动作-肌肉-伤病关系链）。

终局方向：
- **数量**：39 个动作不够，理想覆盖 ~200-300 个常见训练动作
- **质量**：知识库本身的准确性未经人工审核（仅验证了 LLM 生成题目的准确性）
- **时效性**：运动科学在快速更新，知识库需要版本化
- **个性化数据**：用户训练日志、伤病历史、进步曲线——比通用知识更有价值，目前完全没接入

## 二、评测方法

### Q6: 换了 Agentic RAG 架构后怎么做 Evaluation？

评测需要分层，不能只看最终回答。本项目采用三层评测：

**第一层：检索评测（不变）**
P@K, R@K, MRR, NDCG@K——这些指标和普通 RAG 一样，测的是"检索对不对"。消融实验的 A/B/C 三组就是在这层对比的。

**第二层：Agent 决策评测（新增）**
Agentic RAG 引入了 LLM 路由和自评决策，需要额外评测：
- Planner 路由正确率（本项目：53% → 92%）
- 查询改写是否产生了语义漂移（本项目还没测）
- 自评是否终止过早或过晚（本项目还没测）

**第三层：端到端生成评测（新增）**
LLM-as-Judge 打分（准确性/安全性/有用性），测最终回答质量。本项目用 DeepSeek 当裁判，10 题平均 4.90/5。

三层互补：检索好 ≠ 路由对 ≠ 回答好，各自独立归因。

**消融实验设计思路：**
| 组 | 配置 | 目的 |
|----|------|------|
| A (Baseline) | 纯向量检索 | 基线 |
| B (Agentic RAG) | A + LLM 自评 + 查询改写 | 量化 Agentic 增益 |
| C (Full RAG) | B + GraphRAG + 知识库 RRF + 重排序 | 量化全量增益 |

关键发现：
- 中文 trigram 关键词检索对知识库适得其反（P@5 -46%）
- LLM 重排序是中文场景的核心增益（R@5 0.89→0.96）
- 动作名匹配不需要复杂检索，向量语义已足够（三组持平）

### Q7: 测试集是 AI 生成的吗？

**部分是。** 两套数据集：

| 数据集 | 数量 | 来源 | 用途 |
|--------|------|------|------|
| 自建 Golden Dataset | 60 条 | 手工构造，覆盖增肌/减脂/动作分析/伤病/混合 | 动作名检索评测 |
| LLM 生成知识题 | 54 条 | DeepSeek 读 18 篇知识文档后生成，人工审核 | 知识库检索评测 |

LLM 生成流程：DeepSeek 读每篇知识文档 → 生成 3 道问答题 → 人工抽查验证。刚做了抽查：10/10 零幻觉。

面试要点：**LLM 生成 + 人工审核 = 出题人和开发者分离**。手动写测试题会下意识写"系统擅长的题"，LLM 出题打破了这种偏见。

## 三、Agent 框架与记忆架构

### Q8: 通用 Agent 框架具体是做什么的？

结合本项目的实际架构，Agent 框架做三件事：

**1. 任务路由（Planner）**
用户输入 → 判断意图 → 分发到对应 skill。本项目用关键词匹配（SkillRegistry），更通用的做法是 LLM function calling 做动态路由。

**2. 多 Agent 编排（Orchestrator）**
Planner → Retriever → Writer → FactChecker，通过 A2A 消息总线串联。每个 Agent 只做一件明确的事，输入输出结构化。职责分离的好处是每个环节可以独立评测和优化。

**3. 工具调用 + 记忆管理**
Agent 可以调用外部工具（数据库查询、知识图谱推理、API 调用），并在多轮对话间保持上下文。本项目的 ConversationManager 就是记忆模块。

### Q9: 可以详细说一下记忆架构吗？

本项目记忆分两层：

**短时记忆（ConversationManager）**
- 滑动窗口：保留最近 N 轮对话原文
- 摘要压缩：超出窗口的历史轮次用 LLM 压缩成摘要，保留关键信息
- 用途：多轮对话时注入历史上下文，处理"改一下""换成"等指代

**持久记忆（用户画像 + 缓存）**
- UserProfile：身高/体重/训练年限/伤病/目标，跨 session 持久化
- SemanticCache：语义相似度匹配，相同问题不重复调 LLM

**缺失部分**（可主动提出）：
- 长期用户行为记忆——训练偏好、伤病轨迹、进步曲线
- 知识库更新记忆——新研究结论覆盖旧知识

### Q10: 上下文压缩怎么做的？

本项目使用 LLM 摘要压缩：

```
近 3 轮对话 → 保留原文（细节不丢）
3-10 轮 → LLM 摘要（"用户问过深蹲膝盖疼，建议了高脚杯深蹲替代，用户表示有帮助"）
10 轮以上 → 丢弃
```

不是真正的 token 级压缩。更高级的做法：LangChain 的 `ConversationSummaryBufferMemory`，或让 LLM 维护一个"用户状态摘要"作为系统 prompt 注入。

## 四、Prefix Caching

### Q11: Prefix Caching 原理及 Agent 框架怎么保证不破坏缓存？

**原理**：LLM 推理时，如果两个请求的 prompt 前缀相同，KV cache 可以复用前缀部分，跳过重复计算。对于 system prompt 很长 + 多轮对话的场景，节省显著。

**Agent 框架的挑战**：每次调 LLM 可能注入不同上下文（检索结果、工具调用结果、对话历史），前缀很难保持稳定。

**保证策略**：
- system prompt 和工具定义放最前面（不变前缀）
- 动态内容（检索结果、用户输入）放最后
- 多轮对话历史按固定顺序拼接
- 用 conversation ID 做缓存分组，同一个 session 的请求共享前缀

本项目没有显式利用 prefix caching，但如果要用，最简单的方式是把 `PLANNER_SYSTEM` 和 `SkillRegistry` 信息放在 prompt 最前面。

## 五、AI 产品视角

### Q12: 怎么看待 Coze 和 Dify 这类工作流 AI 产品？

**Coze** 的对错：把 Agent 搭建变成拖拽式工作流，降低门槛。但致命问题——用户搭出的东西 90% 是玩具，真正能持续运行的 Agent 需要处理 edge case，可视化工作流恰恰不擅长 edge case 建模。

**Dify** 的对错：定位更偏企业侧（RAG + 工作流 + 监控），比 Coze 务实。但它把 RAG 包装成"一键部署"时，用户不理解 embedding/chunking/rerank 这些关键环节，出了质量问题无法排查。

**共性风险**：这类产品天然倾向于"更长的 prompt"和"更多的 LLM 调用"，导致延迟和成本不可控。真正生产级的 Agent 需要对每个 LLM 调用做成本-收益判断，这是 NoCode 工具目前做不到的。

## 六、OpenClaw

### Q13: 有研究过 OpenClaw 吗？

没有深入研究。OpenClaw 是一个开源的多 Agent 框架，用 Rust 实现，对标 OpenAI Swarm / LangGraph 等编排框架，核心卖点是低延迟和高并发。

如果面试官追问，如实说"没深入研究"比硬编强。可补充：关注过 Agent 框架的性能问题（Python GIL 限制多 Agent 并发），Rust 实现确实有优势。

## 七、Code Agent 文件编辑

### Q14: Claude Code、Cursor 等 Code Agent 是怎么实现文件编辑的？

**核心原理**：不是逐行编辑文件，而是"old_string → new_string"的精确字符串替换。

**为什么不用行号**：行号在编辑过程中会漂移（删一行后面全变了），不可靠。

**为什么不用 sed/正则**：容易匹配到错误位置，尤其是代码中相似的 pattern。

**如果设计 edit 工具**：
```
参数设计：
  file_path: str        // 文件路径
  old_string: str       // 要替换的原文本（必须在文件中唯一）
  new_string: str       // 新文本
  replace_all: bool     // 替换所有匹配（默认 false，确保安全）
  expected_count: int   // 可选，期望匹配次数，防止意外

关键约束：
  - old_string 必须唯一，否则拒绝执行
  - old_string 和 new_string 必须不同
  - 必须先 Read 过该文件（防止在错误的版本上编辑）
```

设计哲学：**安全优先于效率**——宁可拒绝执行也不要改错位置。
