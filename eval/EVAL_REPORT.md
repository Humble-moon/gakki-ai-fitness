## 0. 标签补全（2026-07-17）：为什么要重标，怎么标的

**问题**：2026-07-16 动作库从 39 扩到 106，但 golden set 的 `relevant_doc_ids` 仍基于旧库标注。
新增 67 个动作中的相关动作检索到了也会被判"不相关"，系统性扭曲所有指标。

**补标方法论**（`eval/golden_dataset/relabel_golden.py`）：

1. **穷举判定而非 pooling**：动作库仅 106 个，把全量动作清单（含肌群/器械）交给裁判 LLM
   逐查询判定——避免"用自己的检索器生成候选"的循环偏差（pooling 会让标签偏向检索器已能找到的结果）
2. **异构裁判**：qwen-plus 标注，与检索/生成链路的 DeepSeek 不同厂商，判定独立
3. **只增不删**：原有人工标签全部保留；与 hard_negative_ids 冲突的候选直接丢弃
4. **可审计**：`relabel_diff.md` 逐查询记录新增标签与裁判理由；原文件备份为 `queries.json.bak-20260717`

**结果**：66/80 条查询新增标签，共 +447 个，平均相关集从 ~7 增至 11.1 个。

**对指标的必然影响（先说清楚，防止误读）**：

| 指标 | 方向 | 原因 |
|------|------|------|
| Recall@K | **下降** | 分母（相关集大小）变大；R@10 理论上限只有 0.844 |
| Precision@K | **上升** | 检索到的新动作现在能被正确判为相关 |
| 与 7/10 旧报告 | **不可直接对比** | 标签体系变了，纵向对比无意义 |

### 关键发现 1：Top-5 内三组持平，仍然是真实结论

补标后结论不变：**在 106 个动作的稠密向量空间里，text-embedding-v4 的语义表示对 Top-5 检索已经饱和**。
查询改写无法显著改变向量方向，图谱推理的价值不体现在动作检索的 P@K 上。
HN-Recall@5 = 0.972（补标前 0.903）——97.2% 的"表面相关但危险"的动作被排除在 Top-5 之外，
纯向量空间已能区分"相似"与"危险"。

### 关键发现 2（新）：B/C 组在 K=10 出现可测量的负增益

R@10：A 0.470 → B 0.425 → C 0.417（NDCG@10 同趋势）。**加了组件反而更差**，这是本次补标后
第一次清晰暴露的现象。初步归因（待深挖，诚实标注为假设）：

- **C 组**：知识库补充结果按 title 追加在动作结果之后，挤占了 Top-10 中本属于相关动作的位置——
  知识文档对"动作检索"这个任务是噪声
- **B 组**：Agentic 自评触发改写后，深排名区（rank 6-10）的结果顺序被重排，长尾相关动作被换出

**工程含义**：对动作检索任务，Agentic 循环和知识库融合应该只作用于"检索不足"的场景
（自评不通过才触发），而不是无条件叠加。这个负增益就是"组件不是越多越好"的直接证据。

## 2. Agent 决策质量

### 2.1 Planner 路由正确率

- 总体正确率：**91.2%**（73/80，全量 80 条含能力专项）

**混淆矩阵**：

| True \ Pred | exercise_analysis | fat_loss | muscle_building |
|-------------|-------------------|----------|-----------------|
| exercise_analysis | 44 | 0 | 0 |
| fat_loss | 4 | 10 | 2 |
| muscle_building | 1 | 0 | 19 |

**按类别**：exercise_analysis 100%（44/44）、muscle_building 95%（19/20）、fat_loss 62.5%（10/16）

- 主要错误模式：fat_loss 被路由到 exercise_analysis（4 条）——减脂查询若含具体动作名，
  LLM 语义路由倾向按"动作"理解。方向上偏保守（动作分析比生成计划更安全）

### 2.2 FactChecker 安全审查（29 条 medium/high 样本）

| 指标 | 全体 (n=29) | 伤病子集 (n=21) | 说明 |
|------|------------|----------------|------|
| 召回率（Recall） | 95.2% | **100%** | 危险样本被标记的比例 |
| 漏报率（FNR） | **4.8%**（1 例） | **0.0%** | 危险样本未被标记（最关键指标） |
| 精确率（Precision） | 74.1% | 95.2% | 标记中真正危险的比例 |
| 误报率（FPR） | **87.5%** | 100% | 无害样本被误标为危险 |
| F1 | 0.833 | 0.976 | |

**诚实解读（这一段比 0% 更有面试价值）**：

1. **伤病子集 FNR 保持 0%**——HITL 确定性规则引擎（伤病关键词→禁止动作映射表）对已建模的
   伤病冲突做到了硬性兜底，这部分不依赖 LLM，结果可复现
2. **全体 FNR 4.8%（1 例非伤病漏报）**——漏报发生在规则表未覆盖的非伤病风险场景，
   暴露了规则引擎的边界：规则只能拦截已知模式
3. **FPR 87.5% 是"宁可误报不可漏报"的代价，且第一次被量化**。误报的成本是触发人工审核
   （HITL 流程），不是直接给用户错误建议——在健身安全场景这个 tradeoff 是合理的，
   但如果误报率影响运营成本，下一步应该做的是给规则引擎加白名单而不是放松阈值

## 3.5 RAGAS 生成质量评测（20 条，qwen-plus 裁判）

| 指标 | 均值 | 区间 | 含义 |
|------|------|------|------|
| Context Relevance | **0.988** | 0.750–1.000 | 检索上下文与查询的相关度 |
| Answer Relevancy | **0.793** | 0.671–0.931 | 答案是否切题 |
| Faithfulness | **0.779** | 0.276–0.963 | 答案是否忠于检索上下文 |

### 发现：Faithfulness 的低分集中在"计划生成类"查询

最低的三条（q001 0.276 / q003 0.591 / q016 0.594）全部是"给我一份训练计划"类查询——
生成计划需要模型补充上下文中没有的编排细节（组数/次数/休息/分化安排），这些内容来自
参数化知识而非检索文档，被 Faithfulness 判为"未落地"。知识问答类查询的忠实度普遍在 0.85+。

**诚实解读**：这不完全是缺陷——计划生成本来就需要超出文档的组合创作，但它量化了一个
真实风险：**计划类回答中约 1/4 的陈述没有知识库背书**，这部分内容的正确性完全依赖
基座模型，FactChecker 的安全审查因此更加重要（它拦的正是这部分）。

**方法论备注**：RAGAS 裁判用 qwen-plus（与生成模型 DeepSeek 异构）；3 条小样本预跑时
Faithfulness 只有 0.41，20 条后回升到 0.78——再次说明小样本数字不可信（见 §4-1）。

## 4. 评测局限性（主动声明）

1. **样本量**：基础集 59 条（非伤病）、能力子集各 10 条。所有 ±5% 以内的差异都应视为噪声，
   本报告只主张方向性结论
2. **标签由 LLM 判定**：补标用 qwen-plus 异构裁判 + 只增不删 + diff 可审计，但仍是模型判定，
   未做全量人工复核（已做抽查）。个别广谱查询标签集偏大（q005 有 59 个相关动作），
   会拉低该查询的 Recall 上限
3. **Recall@10 理论上限 0.844**：检索只返回 Top-10，相关集平均 11.1 个，Recall 天花板由标签
   规模决定，不代表系统缺陷
4. **迭代调优的泄漏风险**：Planner/FactChecker 的历史优化是对着本数据集做的，91.2% / FNR 4.8%
   包含一定程度的"记住测试集"成分。严格做法是留 held-out 集（未实施，已列入待办）
5. **运行间波动**：Planner/FactChecker/Agentic 自评含 LLM 调用，跨运行结果有 1-3% 漂移
   （本次 Planner 91.2% vs 上次 95%，部分来自非确定性）
6. **LLM-as-Judge 的宽严差异**：端到端生成质量另见 `E2E_EVAL_REPORT.md`（qwen-plus 异构裁判），
   绝对分数受裁判宽严影响，只有同裁判下的相对比较可信
