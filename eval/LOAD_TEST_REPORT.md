**测试时间**：2026-07-17
**环境**：Windows 单机（本地 docker：PG/Neo4j/Redis/MinIO），uvicorn 单进程，`eval/load_test.py`
**原始数据**：`load_test_results.json`

> 定位：回答"上线后怎么办"。此前评测体系只有离线质量指标（P@K / LLM-as-Judge / RAGAS），
> 没有任何 serving 维度数据——这份报告补上吞吐、延迟分布、流式首响、故障降级四块。

## 1. 应用层吞吐（GET /admin/metrics，无 LLM）

| 并发 | 吞吐 | P50 | P95 | P99 | 错误 |
|------|------|-----|-----|-----|------|
| 1 | 1003 rps | 0.9ms | 1.1ms | 1.6ms | 0 |
| 10 | 947 rps | 9.4ms | 21.2ms | 28.7ms | 0 |
| 50 | 607 rps | 76ms | 212ms | 320ms | 0 |

**结论**：应用层本身能扛 ~600-1000 rps，**系统瓶颈不在 FastAPI/自身代码，在外部 LLM API**。
这正是为什么值得做重试/降级/缓存，而不是优化 Web 框架。

## 2. QA 全链路（POST /api/ask-question，流式 SSE）

| 指标 | c=1 | c=5 | 说明 |
|------|-----|-----|------|
| 首个 SSE 事件 | ~2ms | ~7ms | 进度事件（"正在检索…"）即刻到达 |
| **首个答案 chunk** | **~10.1s** | — | 用户看到第一个字的真实延迟 |
| 总时长 P95 | 19.7s | 18.8s | 完整回答 |
| 错误率 | 0% | 0% | 并发 5 无退化 |

**发现 1：首字延迟 10s 里约 8.6s 花在检索链**。消融实验测得 C-Full 检索 8.6s/查询、
质量增益为零（见 EVAL_REPORT §1）——两份数据交叉印证同一个结论：**Agentic+知识库融合
应条件触发**。若默认走纯向量（0.2s），首字延迟可从 ~10s 降到 ~2s，质量不变。

**发现 2：SSE 进度事件设计让"体感等待"远小于真实延迟**——前端毫秒级收到
"正在检索知识库"等阶段事件，用户不是盯着白屏等 10 秒。这是已经做对的部分。

**发现 3（缺陷）：空 question 会让 SSE 流中途裸断**（服务端 embedding 400 →
连接直接切断，客户端收到 incomplete chunked read 而不是干净的 error 事件）。
API 边界缺输入校验，已列入待修。

## 3. 故障演练：主模型宕机（降级链验证）

**方法**：`LLM_DEFAULT_BASE_URL` 指向死端口模拟 DeepSeek 全挂，
`LLM_FALLBACK_CHAIN=default,judge`（qwen-plus 兜底），发真实 QA 请求。

| 指标 | 正常模式 | 主模型宕机 |
|------|---------|-----------|
| 回答完整性 | 完整 | **依然完整**（67 chunks 由 qwen-plus 送达） |
| 错误率 | 0% | **0%** |
| 总时长 | ~13.5s | **~74-80s（5-6 倍）** |

**结论**：降级链端到端有效——主模型死透用户依然拿到完整回答，服务不中断。
但代价暴露了一个架构缺陷：**没有熔断器（circuit breaker）**。QA 路径含多次 LLM 调用
（自评/改写/生成），每次调用都独立烧完 1s+2s+4s 重试才降级，同一个已知死掉的模型
被反复重试。加熔断（连续 N 次失败后直接跳过主模型，冷却期后半开探测）可把降级模式
时长从 ~78s 压到 ~20s。这是下一步最明确的工程改进。

## 4. 局限性

1. 单机 Windows 本地测试，无网络抖动/跨机房因素；生产部署数字会不同
2. QA 场景样本小（15 请求）——LLM 计费限制；延迟波动 ±2s 属正常
3. 未测长会话内存增长、连接泄漏等 soak 类问题
4. 压测客户端与服务端同机，应用层吞吐数字略乐观
